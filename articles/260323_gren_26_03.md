A new version of the Gren programming language is now available for download. This release is backwards compatible, and brings improvements in parsing Gren code, and to parsers you define on your own.

## What do we mean by 26.03?

We aim to do two releases per year. 26.03 is a marketing term. It refers to the releases of the compiler and core packages made in March 2026.

This release consists of:

* [compiler 0.6.4](https://github.com/gren-lang/compiler/releases/tag/0.6.4)
* [gren-lang/core 7.4.0](https://packages.gren-lang.org/package/gren-lang/core/version/7.4.0/overview)
* [gren-lang/compiler-common 1.0.0](https://packages.gren-lang.org/package/gren-lang/compiler-common/version/1.0.0/overview)
* [gren-lang/compiler-node 4.0.0](https://packages.gren-lang.org/package/gren-lang/compiler-node/version/4.0.0/overview)

As you can infer from the semantic versions, this release retains backward-compatibility at the language level.

## Gren parser as a package

In the new [gren-lang/compiler-common](https://packages.gren-lang.org/package/gren-lang/compiler-common/version/1.0.0/overview) package,
you’ll find a lot of the implementation of the Gren compiler. Perhaps the most interesting thing in this package is the language parser.

This is a common package, so you can write Node.js- or browser-applications that parse Gren code. Like this:

```gren
module Test exposing (simpleModule)

import Compiler.Parse.Module as Module
import Compiler.Parse.Context as Context
import String.Parser.Advanced as Parser

simpleModule =
  Parser.run Module.parser Context.empty
    """
    module Test exposing (value)

    value : Int
    value = 42
    """
```

The compiler itself still makes use of the Haskell-based parser. We thought it’d be best to wait on swapping out the parser until we have removed any unknown bugs and taken another pass on the error messages it creates.

[Gilbert Ramirez](https://github.com/gilramir) is currently working on a formatter that takes the result of this parser and turns it into pretty-printed text. In combination, this will allow you to generate code as well as format the code you already have.

The formatter didn’t make it in as part of this release, but will be released when it’s ready.

## New parser module

[gren-lang/parser](https://packages.gren-lang.org/package/gren-lang/parser/version/6.2.1/overview) is now deprecated. To replace it, we’ve added two new modules to [gren-lang/core](https://packages.gren-lang.org/package/gren-lang/core/version/7.4.0/overview): [String.Parser](https://packages.gren-lang.org/package/gren-lang/core/version/7.4.0/module/String.Parser) and [String.Parser.Advanced](https://packages.gren-lang.org/package/gren-lang/core/version/7.4.0/module/String.Parser.Advanced).

These modules improve on the original parser package in the following ways:

* Custom operators have been replaced by the [keep](https://packages.gren-lang.org/package/gren-lang/core/version/7.4.0/module/String.Parser#keep) and [skip](https://packages.gren-lang.org/package/gren-lang/core/version/7.4.0/module/String.Parser#skip) functions.
* Instead of a context, you have a more flexible [payload](https://packages.gren-lang.org/package/gren-lang/core/version/7.4.0/module/String.Parser.Advanced#getPayload) that you can fill with whatever you want. Like the current indentation level or user comments.
* A function to [pretty-print errors](https://packages.gren-lang.org/package/gren-lang/core/version/7.4.0/module/String.Parser.Advanced#renderDeadEnds), including a display of where in the code the error occurred.
* Removed buggy and duplicate-functions
* Fixed several bugs

## Misc changes

* Gren code in the compiler now compiles with Gren 0.6.x
* You can now use [Bytes](https://packages.gren-lang.org/package/gren-lang/core/version/7.4.0/module/Bytes) in flags and in ports
* Inexhaustive pattern matches involving records no longer compile
* Local dependencies, which themselves have local dependencies, now compile as long as the root project references those same dependencies.
* Fixed typos in a few error messages

## What’s next

Our eyes now turn toward the future.

As already mentioned, [Gilbert Ramirez](https://github.com/gilramir) keeps working on the formatter. People have been asking for formatting for a long time, and it’ll be interesting to see what people will do once they’re able to generate Gren code.

[Justin Blake](https://github.com/blaix) is still working on the new package registry, which will let us and the community host projects somewhere other than Github.

[Robin Heggelund Hansen](https://github.com/robinheghan) will start working on parametric modules and swapping out the Haskell-based parser with this new Gren-based one. It might just be easier to do the canonicalization re-write simultaneously, but we’ll cross that bridge once we get there.
